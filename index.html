Can a language model build a working game from one sentence?

An independent, reproducible benchmark: 60 large language models across 8 vendors, each given the same one-line prompt and asked to generate a complete, playable Sudoku game — then machine-tested for whether it actually renders, is logically valid, and can be played to a win. Every generated game below is playable in your browser.

Abstract

We measured how reliably large language models can turn a single natural-language prompt into a complete, correct, fully-playable browser game. Using Sudoku as the probe — because its correctness is objectively machine-verifiable — we gave 60 models from 8 vendors the identical prompt and evaluated each generated game on three gates: it must deliver, encode a valid puzzle that can be driven to a recognized win, and visually render a usable board. Of the 60, 41 produced a working game and 23 produced a board that was both logically correct and cleanly rendered. We report generation time, estimated cost, feature richness and a per-vendor reliability breakdown, and we publish every generated game so you can try them yourself. A central finding: logic-only checks give false positives — a model can compute a correct solution yet render a broken board, or render a convincing board around an invalid puzzle — so both logical and visual verification are required.

1. Methodology

1.1 The probe

Every model received the exact same prompt — deliberately short and casual, to mirror how a real user types:

create a sudoku game with jazz music.

Sudoku is an ideal probe because a correct result is checkable by machine on three independent axes: it must render a grid with given digits, the encoded puzzle must have a valid solution, and the game must recognize a completed win. It also exercises layout, logic and interactivity at once.

1.2 Models

We enumerated 60 current text models across 8 vendors (full roster in the appendix): Anthropic, OpenAI, Google, xAI, DeepSeek, Mistral, MiniMax and Alibaba (Qwen). The only variable was the text model that wrote the game; everything else — including the AI-generated jazz soundtrack — was held constant, so differences are attributable to the model.

1.3 Generation

Each model was given the prompt and produced a complete, self-contained game, which PlayPrompt turned into a single playable page (gameplay code plus an AI-generated jazz track). We recorded generation time and token usage for every model.

1.4 Evaluation — three gates

Every delivered game was tested in a real headless browser (Playwright / Chromium):

  • Delivery — did the model return a complete game at all?
  • Logic — load the game, extract the puzzle and its intended solution, verify it is a valid Sudoku, then drive the full correct solution into the board through the game's own controls and confirm the game registers a win.
  • Visual — screenshot the started game and inspect the pixels for a real grid, visible 3×3 box divisions, printed given numbers, and a non-garbled layout.

Each game was classed clean (correct 9×9, rendered well, driven to a win), reduced (a valid but non-standard variant), flawed (renders but invalid puzzle / broken input / nothing to play) or broken (no usable board).

1.5 Cost & features

Estimated text-generation cost = recorded token usage × each model's published per-token list price (asset costs excluded, to isolate the text model). Feature counts are the number of distinct interactive elements catalogued per game and are approximate.

5. Discussion

  • Three gates, all load-bearing. Of 41 delivered games, only 23 were both correct and cleanly rendered. Failures split across invalid puzzles, broken input, and broken rendering — so evaluating generated software means driving it and inspecting the pixels, not just reading the code or trusting internal state.
  • Cost did not predict quality. The most expensive run produced an invalid puzzle; the cheapest clean game cost a fraction of a cent.
  • Coding-tuned models over-perform for their price. The fast, cheap clean games come disproportionately from vendors' coder variants.
  • Spec interpretation varies. Some models read "Sudoku" as a simplified 4×4 variant rather than the standard 9×9 — valid, but not what most users expect from one ambiguous sentence.
  • Top tier is reliable end-to-end; several smaller or older models deliver something that looks right but fails on play or on render.

6. Limitations

  • Single prompt, single run per model. LLM output is non-deterministic; a re-run may differ. This is a snapshot, not a leaderboard.
  • Point in time (2026-06-23). Model line-ups and quality move quickly.
  • Costs are estimates from published list prices applied to recorded tokens; actual pricing varies by region, tier and promotions.
  • Feature counts are approximate and not strictly comparable across games.
  • Solution uniqueness was not uniformly enforced. A proper Sudoku has exactly one solution; our bar for "clean" was a valid, winnable board. A few generators produce valid but non-unique puzzles (flagged ⚠); the same soft defect may exist undetected elsewhere.
  • Visual QA inspects the board on load / after start, not every difficulty level or animation.

7. Reproducibility

  1. Enumerate the current public text models for each vendor (the roster below).
  2. Give every model the same prompt — create a sudoku game with jazz music. — and capture the generated game, its generation time and token usage.
  3. For each game, run a headless-browser check that renders the board, validates the puzzle, drives the full solution to a recognized win, and screenshots the result.
  4. Estimate cost as recorded tokens × the model's published list price.

Every game in this study was created with PlayPrompt — an agentic AI platform that turns a single text prompt into a complete, ready-to-play browser game, generating the gameplay, the music, and, when needed, the visual assets. In this study we swapped only the underlying text model; the music, assembly and packaging into a playable page were PlayPrompt throughout.
